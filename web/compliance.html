ZK Set Proofs

Prove legitimacy without revealing identity.

Both proof types are generated client-side. Any counterparty — exchange, auditor, regulator — can verify them without learning anything about the user.

Membership Proof
C ∈ A
Proves the commitment belongs to the association set of clean, screened commitments. Verifiable by any counterparty. Reveals nothing about the commitment's origin, amount, or user identity.
Non-Membership Proof
C ∉ E
Proves the commitment does not appear in the exclusion set of flagged or sanctioned commitments. Implemented via sparse Merkle tree non-membership proof. Zero information leakage beyond the binary yes/no result.
How Verification Works
  • User generates proof client-side — no server involvement
  • Proof is a small ZK artifact (<1KB)
  • Counterparty runs the verifier against the proof and public inputs
  • Verifier learns only: valid ∈ {true, false}
  • No user identity, address, balance, or transaction history exposed
  • Proof can be submitted to any verifier — on-chain or off-chain
  • Temporal validity: verifier also checks valid_at against flagging timestamp
Temporal Validity

Flagging events do not reach backward.

A commitment that was clean at time t remains provably clean for proofs generated before it was flagged. This prevents retroactive compliance violations while ensuring newly flagged commitments cannot be used going forward.

t = 0
Deposit
Commitment C created and inserted into Merkle tree. Clean — not in exclusion set E.
t = 100
Proof Generated (valid_at = 100)
User generates withdrawal proof. valid_at = 100 recorded as public input. C ∉ E at this moment — proof is valid.
t = 200
Flagging Event
C added to exclusion set E at time t' = 200. This is recorded with the flagging timestamp on-chain.
Verification
Temporal Check
Proof with valid_at = 100 passes (100 < 200). New proof with valid_at = 300 fails (300 > 200). Retroactive invalidation does not occur.
// Temporal validity rule
valid = proof.valid_at < flag_timestamp(C)

// Before flagging: valid_at=100, flag_time=200
100 < 200 → valid ✓

// After flagging: valid_at=300, flag_time=200
300 < 200 → invalid ✗

// valid_at is a public input in the ZK proof
// Cannot be forged or altered without invalidating proof
Governance

Authority is a parameter, not a constant.

The flagging authority is stored in the ComplianceRegistry contract as a governance-controlled parameter. It is never hardcoded. Changes require a timelock delay. Jurisdictional adaptation without contract upgrades.

Parameter
Flagging Authority
On-chain address with permission to add commitments to the exclusion set. Set at deployment, changeable via governance + timelock.
Constraint
Timelock Delay
Changes to the flagging authority require N blocks delay (configurable at deployment). Prevents sudden unilateral authority transfers.
Design Intent
Jurisdictional Flexibility
Different deployments can specify different flagging authorities — EU regulator, US Treasury list, community-curated — without contract code changes.
What the Flagging Authority Can and Cannot Do
Can
  • Add commitment hashes to the exclusion set
  • Add commitment hashes to the association set
  • Update list provider data sources
Cannot
  • Access or move user funds
  • Deanonymize users or reveal transaction history
  • Retroactively invalidate proofs generated before flagging
  • Change the core protocol without timelock + governance vote
  • Act as a single point of failure — authority change requires timelock
