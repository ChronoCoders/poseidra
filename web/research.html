Research Paper

Poseidra: Formal
Specification

A full academic treatment of the protocol — adversary model, cryptographic primitives, privacy definitions, and security proofs.

Traditional transaction obfuscation systems rely on heuristic mixing techniques that are increasingly ineffective and legally vulnerable. This paper proposes Poseidra, a modern architecture for privacy-preserving financial transactions based on non-custodial design, zero-knowledge cryptography, and selective disclosure. We present a formal five-class adversary model, specify concrete cryptographic primitives with domain separation, define a dynamic denomination-specific anonymity set with formal privacy bounds, and introduce a compliance layer based on association and exclusion sets with temporal validity.

Contributions

What this paper adds.

01
Five-Class Adversary Model
Explicit modeling of MEV block builders and cross-layer correlation — threat classes absent from every comparable protocol specification.
02
Dynamic Anonymity Set
Denomination-specific N_t evaluated at proof generation time, not deposit time. Formal bound: Adv_link ≤ 1/N_t + ε.
03
Domain Separation
All Poseidon hash constructions carry distinct domain tags. Cross-construction collision attacks are cryptographically precluded.
04
Compliance as Core Architecture
Association and exclusion set proofs are first-class protocol components, not post-deployment add-ons.
05
Temporal Validity
Compliance proofs carry valid_at timestamps. Flagging events do not retroactively invalidate previously generated proofs.
06
PLONK/KZG Throughout
No Groth16 dependency. Universal trusted setup applies across all circuit versions. No per-circuit ceremony required.
Formal Definitions

Precisely stated. Not claimed.

Definition 1 — Anonymity Set
Let N_t denote the size of the anonymity set at time t, defined as the number of valid, unspent commitments with equal denomination and a valid root window at the moment a withdrawal proof is generated.
Definition 2 — Privacy
A scheme satisfies privacy if any PPT adversary has advantage Adv_link ≤ 1/N_t + ε, where ε is negligible in the security parameter. The anonymity set is dynamic; privacy is evaluated at proof generation time.
Definition 3 — Soundness
No adversary can produce an accepting proof without knowledge of a valid secret and nullifier corresponding to a commitment in the Merkle tree. No adversary can reuse a nullifier across two withdrawals.
Definition 4 — Unlinkability
Given full on-chain data and timing information, no adversary can correlate deposit and withdrawal flows beyond negligible probability, provided N_t is sufficiently large.
Informal Security Argument
  • Poseidon preimage resistance ensures commitment hiding
  • Merkle inclusion proof ensures set membership
  • Nullifier uniqueness prevents double spend
  • PLONK zero-knowledge property ensures no private input leakage
  • Domain separation prevents cross-construction attacks
  • Root validity window bounds proof age and anonymity set drift
Related Work

Where Poseidra fits.

Tornado Cash
Deprecated
Pedersen commitments, Groth16, Merkle trees. No compliance layer. Trusted setup per pool. Regulatory action 2022 demonstrated the legal fragility of privacy-first non-compliant designs.
Zcash
Dedicated Chain
Pioneered zk-SNARKs for private transactions (Sapling). Strong privacy guarantees. Not designed for composability with EVM smart contract ecosystems.
Railgun
Active
Closest competitor. Uses Groth16 with 54 circuits, UTXO model, community broadcasters. Compliance (PPOI) added post-deployment as an independent partner project — not core architecture.
Aztec Protocol
Active
Programmable privacy via zkRollups. Supports private DeFi. Higher system complexity and distinct execution environment limit direct EVM composability.
Privacy Pools
Closest Conceptual Basis
Formal paper by Buterin et al. introducing association sets for selective disclosure. Poseidra adopts and extends this model with explicit exclusion set proofs, temporal validity, a governance-parameterized flagging authority, and a full production engineering specification. Privacy Pools remains a theoretical specification without a production implementation.
