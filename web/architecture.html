System Stack

Five layers. Zero custody.

Smart Contracts · Solidity + Foundry
Poseidra.sol — deposit, withdraw, nullifier tracking
Accepts fixed denomination deposits. Verifies PLONK proofs on withdrawal. Enforces root validity window.
On-Chain
ComplianceRegistry.sol
Association set A, exclusion set E. Governance-parameterized flagging authority.
Generated
Verifier.sol
PLONK/KZG verifier. Generated from final circuit via snarkjs. Never handwritten.
ZK Circuits · circom → halo2
withdraw.circom
Commitment + nullifier + Merkle inclusion. ~2,300–2,600 constraints.
Proof Engine · Rust
Prover
Witness generation, Poseidon computation, PLONK proof generation. Target: <2s.
Backend · Rust + Axum
Indexer API + Merkle State
Indexes deposits, maintains local Merkle tree, serves paths for witness generation. PostgreSQL + Redis.
Compliance · Rust
Set Proofs + Risk Scoring
Membership / non-membership proofs. Off-chain screening API with caching.
Relayer Network · Rust
Commit-Reveal Fee · Multi-Instance · Direct Fallback
Semi-trusted. Cannot steal funds. Cannot break cryptographic privacy. Multiple instances required in production.

Cryptographic Primitives

Every choice justified.

Primitive Choice Rationale
Hash Function Poseidon (t=3) SNARK-friendly; ~90 constraints per invocation vs thousands for SHA-256
Commitment Poseidon(domain_C, secret, nullifier) Binding and hiding; domain_C prevents cross-construction collision with nullifier hash
Nullifier Poseidon(domain_N, nullifier, chain_id, contract) chain_id + contract binding prevents cross-chain replay and cross-instance collision
Merkle Tree Binary, Poseidon, depth 20 ~1M leaf capacity; all hashes SNARK-native; consistent with prover and contracts
Proof System (L1) PLONK + KZG Universal setup; no per-circuit trusted setup; L1 pairing-compatible verifier
Proof System (L2) Halo2 / IPA No trusted setup; supports recursive aggregation; targets future L2 scaling
Excluded Groth16 Requires trusted setup per circuit — architecturally unacceptable for an evolving system

Trust Assumptions

What you must trust. What you don't.

Trusted
  • Soundness of the PLONK proof system under the KZG assumption
  • Correct circuit implementation — no under-constrained signals
  • Poseidon parameters identical across circuit, prover, and verifier
  • Perpetual Powers of Tau trusted setup for universal reference string
Semi-Trusted — Relayer
  • Can censor individual transactions
  • Cannot steal funds — held by non-custodial contract
  • Cannot break cryptographic privacy
  • Mitigated by: multiple instances, direct withdrawal fallback
Never Required
  • Trust in any relayer to handle funds honestly
  • Trust in any centralized compliance authority with hardcoded power
  • Per-circuit trusted ceremony (Groth16 excluded for this reason)
  • Trust in any frontend operator to not censor withdrawals
